## Source-rigor pattern (follow on every content/data write)

This is the verified-good YMYL workflow. A single wrong claim in a YMYL article damages site credibility more than 10 correct articles help it. Follow this process on every article, data file, or claim that could affect a reader's health or money decisions.

### Step 1 — primary-source verification BEFORE drafting

- For every numeric claim (trial result, FDA label percentage, drug price, insurance coverage rate), fetch the primary source directly via `WebFetch` before you write a single word of the article body.
- **Primary sources only:** PubMed abstracts (`pubmed.ncbi.nlm.nih.gov/{PMID}/`), NEJM/JAMA/Lancet/Diabetes Care paper abstracts, FDA label PDFs on `accessdata.fda.gov`, FDA press announcements on `fda.gov`, CMS guidance, KFF reports, peer-reviewed meta-analyses.
- **NOT primary sources:** blog posts, PR Newswire, news aggregators, Wikipedia, drugs.com, GoodRx blog, competitor sites, chatbot summaries.
- If a number is NOT in the abstract, do NOT fabricate it. Hedge with qualitative language or omit. The YMYL cost of making something up is infinitely worse than leaving a gap.

### Step 2 — draft using only verified numbers

Use the established registry pattern:
- `src/lib/research.ts` — add entry to `RESEARCH_ARTICLES` and (for Spanish) to `SPANISH_RESEARCH_SLUGS`
- `src/app/research/[slug]/page.tsx` — article body with inline `[1]`, `[2]` citations referencing a `citations` array
- `src/app/es/research/[slug]/page.tsx` — thin re-export with Spanish metadata and hreflang (for Spanish articles only)
- `src/app/sitemap.ts` — automatically picks up from the registry
- `next.config.mjs` — add a 301 from `/research/[slug]` to `/es/research/[slug]` for Spanish articles

Include a disclaimer footer about this being informational, not medical advice.

### Step 3 — post-draft multi-agent YMYL verification

Dispatch BOTH in parallel via `Agent({subagent_type: "Explore"})`:

1. **Clinical verifier agent** — enumerate every numeric and regulatory claim and ask it to independently re-verify each one. Prompt pattern: "YMYL pre-ship clinical verification. Read [file path]. Verify the following specific claims against primary sources: [enumerated list with expected values]. Flag anything wrong, unsupported, or overreaching. Specifically scrutinize [regulatory claims that go beyond the abstract]. Report in under 500 words, pass/fail per claim."

2. **Spanish linguistic reviewer agent** (for Spanish articles only) — grammar, medical terminology (semaglutida/tirzepatida consistently feminine), register (neutral US Hispanic Spanish), calcos, inverted punctuation, accent marks. Prompt pattern: "Spanish-language medical editor review. YMYL patient-education article for US Hispanic audience. Read [file path]. List issues with exact old_string → new_string replacements. If ready to ship, say so."

### Step 4 — resolve verifier findings, but DON'T trust them blindly

- If the clinical verifier flags a claim as wrong, **independently re-verify** via `WebFetch` against the actual primary source before changing anything. Verifier agents can be wrong too. In this session the Rybelsus A1C verifier was wrong about −1.1 vs my verified −1.4 — the primary source confirmed my number.
- If the verifier flags a claim you CAN'T reach primary sources for (e.g., FDA endpoints 404, paper is paywalled), the YMYL-safe move is to hedge the claim in the article rather than trust secondary sourcing. The Rybelsus oral-semaglutide-for-weight-management claim is an example: I couldn't independently confirm an alleged Dec 2025 approval, so I rewrote the relevant section to direct users to Drugs@FDA for current status rather than making a claim I couldn't verify.
- If the linguistic reviewer flags a decimal-separator convention change, decline if it would create inconsistency with the rest of the article's numeric style.

### Step 5 — ship with an auditable commit message

The commit message should enumerate:
- What sources were verified directly via WebFetch
- What the clinical verifier agent caught and how you resolved it
- What the linguistic reviewer caught and how you resolved it
- Any claims that were deliberately hedged instead of asserted, and why

This creates an audit trail for the site's YMYL posture.
